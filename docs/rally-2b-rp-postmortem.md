# Rally 2B RP Publish Postmortem

Date: April 13, 2026

## Summary

`thomasjvu/rally-2b-rp` trained successfully on the H200 box, but the first publish attempt stopped before export and upload. The model itself was fine. The failure was in the post-training packaging path: the merged local checkpoint did not contain `generation_config.json`, so `inspect` rejected the source repo and `convert` exited early.

After recovery, the published package itself was correct for the Gemma v2 multimodal browser contract, but the autogenerated README mislabeled the repo as a v1 package because the model-card logic inferred versioning from the repo name suffix instead of the manifest capabilities.

## Impact

- Training completed after about 2 hours and 43 minutes.
- No ONNX export, package assembly, or Hugging Face upload happened on the first pass.
- The published README was misleading after recovery:
  - it correctly listed `text`, `image`, `audio`, and `video`
  - it incorrectly described the repo as "stable v1"
  - it incorrectly pointed readers to a nonexistent or irrelevant `-v2` sibling

## Timeline

1. The Rally RP text-SFT dataset was prepared from `Maxx0/Texting_sex` with filtering and normalization.
2. LoRA fine-tuning completed successfully for the Rally 2B Gemma checkpoint.
3. The merged checkpoint was written to `build/phala_gpu_tee/models/rally-2b-rp-merged`.
4. `render-manifest` succeeded using `configs/heretic-to-onnx.gemma4-e2b-heretic-ara-v2.yaml`.
5. `convert` failed during `inspect` because the merged source repo was missing `generation_config.json`.
6. A follow-up fix added `generation_config.json` to merged checkpoints and the existing trained checkpoint was recovered without retraining.
7. After publish, the README inconsistency was traced to repo-name-based version-note logic in `tools/heretic_to_onnx/publish_hf.py`.

## Root Causes

### 1. Merged checkpoints were missing `generation_config.json`

The trainer saved:

- model weights
- tokenizer files
- merged 16-bit checkpoint weights

but it did not explicitly persist `model.generation_config` into the merged export directory. The ONNX packaging pipeline treats `generation_config.json` as a required source asset for Gemma checkpoints, so the pipeline correctly stopped.

### 2. README version notes were keyed off the repo name, not the runtime contract

The model-card generator treated any repo whose name did not end in `-v2` as a v1 package. That assumption held for the direct browser exports, but it broke for RP repos because:

- `rally-2b-rp` intentionally targets the Gemma v2 manifest
- the repo name does not include `-v2`
- the package still contains the full multimodal browser contract

As a result, the README combined:

- correct capability bullets from the manifest
- incorrect v1/v2 prose from the repo-name heuristic

## Contributing Factors

- The RP repos intentionally kept shorter unsuffixed names.
- The failure surfaced only after the long training job had already finished.
- The one-click script is end-to-end, so the user saw "it trained" and "it did not publish" in the same run without a separate recovery step.

## Fixes Applied

### Trainer fix

Commit `3a2ba97` updates [scripts/train_rally_unsloth.py](/Users/area/heretic/scripts/train_rally_unsloth.py) to save `generation_config.json` into the merged checkpoint directory after `save_pretrained_merged(...)`.

### README/model-card fix

The model-card generator now treats unsuffixed repos with enhanced multimodal capabilities as using the v2 browser contract. For repos like `rally-2b-rp`, the README now says that the repo itself is the multimodal variant instead of claiming it is v1 and pointing to a synthetic `-v2` sibling.

## Lessons

- Required source assets for browser packaging should be validated immediately after merge, not only when the full convert pipeline starts.
- Version labeling should be derived from manifest/runtime capabilities first and repo naming second.
- RP model naming and browser runtime versioning are separate concerns and should not be conflated in autogenerated docs.

## Follow-Up

- Keep RP repos on the v2 Gemma/Qwen export manifests when they are intended to retain multimodal browser support.
- Prefer README generation rules that reflect `manifest.modalities` and architecture-specific capability supersets.
- Use `publish-model-card-hf` for README-only corrections so docs can be updated without repackaging or reuploading the full model.
